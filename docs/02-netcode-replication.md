# Aurora — Netcode & Replication Specification

> Version 0.1 (design). The replication model is a *language feature*, not a
> library: `@replicated`, `@interp`, `@predict`, and `@rpc` are checked and
> code-generated by the compiler against the ECS (see grammar spec §4.3, §6).

Design north star: **the author writes gameplay once; the compiler generates
serialization, delta-compression, interpolation, client-side prediction, and
server reconciliation.** Client and server run the *same* functions over the
*same* `Vec3`/`Transform` math — structurally preventing the coordinate-space
drift class of hitreg bugs.

---

## 0. Topology & authority model

Aurora targets **server-authoritative client-server** as the default (the right
default for a competitive milsim shooter). Listen-server (host is also a client)
and dedicated-server are the same code path; P2P/rollback is a selectable mode for
small-player-count games (§9).

```
authority ∈ { server, client(owner), shared }
```

- `server` — the server is the single source of truth. Clients predict & display.
- `client(owner)` — the owning client is authoritative (e.g. cosmetic-only state,
  or trusted-client modes). Other clients receive replicated copies.
- `shared` — CRDT-style last-writer-wins with server tiebreak (rarely used).

Every `@replicated` component declares its authority (default `server`).

## 1. Tick model & time

```
const TICK_HZ = 64            // simulation rate (configurable per project)
struct Tick(u32)              // monotonic simulation tick number
struct Time { dt: f32, tick: Tick, alpha: f32 }   // alpha = render interpolation factor
```

- The **simulation** runs at a fixed `TICK_HZ` on both server and predicting
  clients (deterministic-friendly fixed timestep).
- **Rendering** runs at display rate; the renderer interpolates between the two
  most recent simulation states using `Time.alpha` (the classic
  fixed-update/variable-render split).
- Each replicated update is stamped with the `Tick` it was produced on. Ticks are
  the common clock for prediction, reconciliation, and lag compensation (§6, §7).

## 2. Declaring replicated state

### 2.1 Component-level attributes
```aurora
@replicated(authority: .server)
component PlayerState {
    @interp           pos: Vec3        // smoothly interpolated on remote clients
    @interp(.slerp)   rot: Quat        // interpolation strategy override
    @predict          vel: Vec3        // predicted locally, reconciled vs server
    health: f32                        // plain replicated (server-authoritative)
    @quantize(0.01)   stamina: f32     // sent as fixed-point, 1cm-style precision
    @noreplicate      fx_timer: f32    // local-only, never sent
}
```

Field attribute semantics:
- `@interp[(.lerp|.slerp|.hermite)]` — remote clients buffer recent values and
  interpolate for smooth motion (§5). `.hermite` uses velocity for spline interp.
- `@predict` — the owning client simulates this field locally each input tick and
  reconciles against authoritative server snapshots (§6).
- `@quantize(step)` — serialize as integer multiples of `step` (bandwidth + your
  10cm-voxel world maps naturally to `@quantize(0.1)`).
- `@noreplicate` — excluded from the wire format entirely (local cosmetic state).
- Default (no attr) — replicated as-is, no interpolation, snapped on receipt.

### 2.2 What the compiler generates per `@replicated` component
For each replicated component the compiler emits:
1. A **bit-packed serializer/deserializer** keyed by a stable `ComponentId`.
2. A **delta encoder** comparing the field set against a per-client baseline.
3. A **dirty mask** (one bit per replicated field) so only changed fields go out.
4. For `@interp` fields: a client-side **history ring buffer** + interpolator.
5. For `@predict` fields: a client-side **input/prediction history** for replay.

No hand-written `serialize()`/`deserialize()` ever. Adding a field to a component
automatically extends the wire format (versioned, §8.3).

## 3. Wire format & bandwidth

### 3.1 Packet structure
```
Packet ::= Header Channel*
Header ::= proto_ver:u8  client_tick_ack:Tick  server_tick:Tick  flags:u8
Channel::= channel_id:varint  payload_len:varint  payload:bytes
```
Channels (each independently reliable/ordered or not):
- `STATE`     — unreliable, sequenced: snapshot deltas (newest wins, drops OK).
- `EVENTS`    — reliable, ordered: spawns/despawns, one-shot gameplay events.
- `RPC`       — reliable or unreliable per declaration (§4).
- `INPUT`     — unreliable, sequenced: client → server command frames.
- `CONTROL`   — reliable: handshake, time sync, congestion signals.

Transport: UDP with a thin reliability layer (ack/nack, retransmit only for
reliable channels) — same niche as ENet/renet, generated and built in.

### 3.2 Snapshot delta encoding (`STATE`)
- The server keeps, per client, the **last tick that client acknowledged**.
- Each outgoing snapshot is a **delta vs the acked baseline**: only entities in the
  client's interest set (§4.4 below... see Interest Management §10) whose dirty mask
  is nonzero are included; only dirty fields within those.
- Field encoding:
  - bools → 1 bit
  - `@quantize(step)` floats → ranged integer, bit-width from declared range
  - enums → `ceil(log2(variant_count))` bits
  - `VecN`/`Quat` → per-component quantized; quats sent as smallest-three (3×N bits + 2-bit largest index)
  - everything else → natural width
- The bitstream is byte-aligned only at channel boundaries; everything inside is
  packed to the bit.

### 3.3 Determinism note
Quantization is applied **before** the server simulates with the value where it
matters (or the server stores both), so the client and server agree on the exact
quantized value — preventing prediction error purely from rounding.

## 4. RPCs (remote procedure calls)

```aurora
@rpc(.server, reliable)
fn fire(dir: Vec3) {
    // Runs ON THE SERVER. `self` is the calling player's entity (authority-checked).
    let origin = self.muzzle_world_pos()
    let hit = physics.raycast_at_tick(origin, dir, self.cmd_tick)   // lag-comp, §7
    if hit { hit.entity.get_mut::<Health>().hp -= self.weapon.damage }
}

@rpc(.client_owner, unreliable)
fn play_local_muzzle_flash(seed: u32) { spawn_fx(.muzzle, seed) }

@rpc(.multicast, unreliable)            // server -> all interested clients
fn on_explosion(pos: Vec3, radius: f32) { spawn_fx(.explosion, pos, radius) }
```

`@rpc(target, mode)`:
- `target ∈ { .server, .client_owner, .multicast, .client(EntityId) }`
- `mode ∈ { reliable, unreliable }` and optionally `ordered`.

Compiler responsibilities:
- Generates the argument (de)serializer and a stable `RpcId`.
- Inserts an **authority check**: a `@rpc(.server)` invoked by a client is routed
  to the server and executed in the server's authority context; a client cannot
  execute it locally on others. Calling a `.server` RPC on the server runs inline.
- Validates that RPC argument types are serializable (no `&`/`~`/`rc`/`dyn`).
- Stamps the call with the caller's `cmd_tick` for lag compensation.

## 5. Interpolation (remote/non-predicted entities)

For each `@interp` field on a remote entity, the client keeps a small ring buffer
of `(tick, value)` samples and renders at a **fixed interpolation delay** behind
the latest received tick:

```
render_time = latest_server_time - interp_delay
interp_delay = max(2 ticks, jitter_estimate * k)     // adaptive
```

- Renderer finds the two samples straddling `render_time` and blends:
  - `.lerp` for positions/scalars, `.slerp` for `Quat`, `.hermite` when a paired
    velocity field exists (smoother for fast movers).
- If the buffer underruns (packet loss), it **extrapolates** for up to `N` ticks
  using last known velocity, then freezes — bounded so a lost player doesn't
  teleport across the map.

This is per-field and automatic; the author only writes `@interp`.

## 6. Client-side prediction & server reconciliation

The owning client must feel zero-latency for its own avatar. Aurora generates the
predict/reconcile loop for `@predict` fields:

### 6.1 Client loop (per input tick)
```
1. Sample input -> Command { tick, buttons, move_axis, view_dir }
2. Append Command to pending-input buffer; send on INPUT channel.
3. Run the SAME movement system locally on the owned entity using Command,
   advancing @predict fields. Store predicted state keyed by tick.
```

### 6.2 On authoritative snapshot arrival (server says: "at tick T your state was S")
```
1. Compare server state S(T) to the client's stored predicted state P(T).
2. If |S(T) - P(T)| <= tolerance: accept, done (no visible correction).
3. Else (misprediction):
   a. Snap the owned entity's @predict fields to S(T).
   b. REPLAY every buffered Command with tick > T through the same movement
      system, fast-forwarding back to the present tick.
   c. The corrected present state is what renders (optionally smoothed over a few
      frames to hide the pop).
```

### 6.3 Why this is bug-resistant
The movement system is **one function**. The server runs it authoritatively; the
client runs the identical compiled code for prediction and for replay. There is no
"client formula" vs "server formula" to drift apart — the exact failure mode behind
the hitreg drift you flagged. The compiler additionally forbids `@predict` movement
systems from reading non-replicated, non-deterministic inputs (wall-clock time,
RNG without a replicated seed) and errors if they do (§8.1 determinism contract).

## 7. Lag compensation (server-side hit validation)

For hitscan/projectile validation the server **rewinds** to the tick the firing
client actually saw:

```
1. The server keeps a ring buffer of recent collider transforms per tick
   (position history, ~1s window).
2. An incoming fire RPC carries cmd_tick (the client's view tick).
3. server_view_tick = cmd_tick adjusted by the client's smoothed RTT/2 + interp_delay
   (the server reconstructs exactly what geometry the client was looking at).
4. raycast_at_tick(origin, dir, server_view_tick) tests against the REWOUND
   collider positions, not the present ones.
5. Apply damage on hit; replicate the result.
```

- The rewind window is capped (e.g. 250 ms) to bound "shot around a corner"
  unfairness; configurable per project.
- `physics.raycast_at_tick` is a builtin; the per-tick collider history is
  maintained automatically for any entity with a `@replicated` `Transform` +
  collider. This directly serves your milsim hitreg requirement.

## 8. Determinism & safety contracts

### 8.1 Determinism contract for predicted systems
A system that writes `@predict` fields is checked to be **deterministic**:
- May read: the entity's components, the current `Command`, replicated state,
  `comptime` constants, and a **replicated RNG** (`Rng` seeded from a replicated
  seed so client and server draw the same numbers).
- May **not** read: wall-clock time, unsynced RNG, frame-rate-dependent values,
  or non-replicated mutable globals. Violations are compile errors.

This makes prediction replay (§6.2) reproduce the server bit-for-bit (modulo
float determinism, §8.2).

### 8.2 Float determinism
Cross-platform float determinism is opt-in via `@deterministic` on the simulation
module, which: pins to a fixed-point or soft-float path for the flagged math,
disables FMA contraction, and forbids transcendental functions without a
deterministic implementation. Off by default (most games don't need cross-platform
lockstep); required for P2P rollback (§9).

### 8.3 Wire versioning & validation
- Each `@replicated` component and `@rpc` gets a content hash of its field
  layout/signature, exchanged at handshake. Mismatch → clear "client/server schema
  mismatch" error rather than silent corruption.
- All deserialization is bounds-checked; a malformed packet drops the packet, never
  UB (the generated readers are safe code, no `unsafe`).
- The server treats all client input as untrusted: authority checks (§4), input
  rate-limiting, and value clamping are generated/enforced.

## 9. Alternate mode: deterministic lockstep / rollback (P2P)

Selectable for small-player-count fighting/RTS-style games:
- Only **inputs** are exchanged; every peer simulates deterministically (`@deterministic`
  required, §8.2).
- On receiving a remote input for a past tick, the engine **rolls back** to that
  tick and re-simulates forward (GGPO-style) using the same replay machinery as
  client prediction (§6.2) — the two share one implementation.
- Tradeoff surfaced to the author: requires full determinism and bounded state for
  fast save/restore; not suited to large open worlds.

---

## 10. Interest management (scaling player counts)

The server doesn't send every entity to every client:
- **Spatial grid / AABB queries** determine each client's interest set (entities
  within view/relevance radius). Built on the same physics broadphase.
- Per-component **priority & rate**: `@replicated(priority: .high, rate: 64)` vs
  low-priority distant cosmetic state sent at a lower Hz; the snapshot budget per
  packet is filled by descending priority (so you degrade gracefully under
  bandwidth pressure instead of dropping the whole snapshot).
- Spawn/despawn events for entities entering/leaving interest go on the reliable
  `EVENTS` channel so clients never miss create/destroy.

## 11. End-to-end example

```aurora
@replicated(authority: .server)
component Avatar {
    @interp pos: Vec3
    @interp(.slerp) view: Quat
    @predict vel: Vec3
    health: f32
    @quantize(0.1) ammo: f32        // 10cm-world-style fixed point
}

struct Command { tick: Tick, mv: Vec2, view: Quat, fire: bool }

// Runs on server (authoritative) AND on owning client (prediction) AND on replay.
@predict_system
system move_avatar(cmd: Command, dt: Time) {
    for (a, e) in query<&mut Avatar, Entity> where owns(e, cmd) {
        a.vel = accelerate(a.vel, cmd.mv, dt)       // pure, deterministic
        a.pos += a.vel * dt
        a.view = cmd.view
    }
}

@rpc(.server, reliable)
fn fire(dir: Vec3) {
    let hit = physics.raycast_at_tick(self.muzzle(), dir, self.cmd_tick)  // lag-comp
    if hit { hit.get_mut::<Avatar>().health -= 25.0 }
    on_explosion.multicast(hit.point, 1.0)          // tell everyone interested
}

@rpc(.multicast, unreliable)
fn on_explosion(pos: Vec3, r: f32) { spawn_fx(.explosion, pos, r) }
```

What you did **not** write but the compiler generated: every serializer, the
dirty-mask delta encoder, the quantizers, the interpolation buffers for `pos`/`view`,
the prediction history + reconciliation replay for `vel`, the RPC routing +
authority checks, the per-tick collider history for lag comp, and the schema hashes
for version validation.

## 12. Open questions

1. **Snapshot vs. event sourcing for inventory-like state** — large, rarely-changing
   structured state may prefer reliable event deltas over snapshotting; pick a
   per-component `mode: .snapshot | .eventlog`.
2. **Encryption/auth** — DTLS-style handshake vs. app-level; out of scope for v0.1
   but the `CONTROL` channel reserves room.
3. **Bot/replay system** — recording the `INPUT` + initial state stream gives free
   deterministic replays when `@deterministic` is on; spec later.
4. **Float determinism scope** — exactly which intrinsics get deterministic
   implementations; needs a per-target audit.
